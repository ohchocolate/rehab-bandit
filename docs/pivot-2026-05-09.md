## 3. Key Judgment

### Schema Drift Is Real, But Not The Core Blocker

The app is evolving faster than the first bandit spec. This is expected because the project is tied to a real body, not a static benchmark.

However, the correct response is not to chase every new session with a schema rewrite. The current strategy should be:

```text
keep normal bandit-ready check-ins small and stable
mark return-test / PT-transition sessions as out-of-distribution
review arms only after repeated OOD patterns accumulate
```

### The Immediate Learning Blocker Is Task Granularity

The author still wants to learn LinUCB. The plan should preserve the Hard Gate, but split the math into smaller steps:

- theta solve
- mean prediction
- confidence bonus
- arm selection
- update

Each step should be small enough to explain in plain English.

### Better Demo Story

The project is becoming more interesting, not less:

> It is no longer only an ankle rehab recommender. It is a personal training recovery scheduler under body-stage shifts.

That naturally introduces real RL concepts:

- non-stationarity
- distribution shift
- OOD samples
- delayed schema migration

But these concepts should be recorded as decisions first, not rushed into algorithm code.

## 4. Decisions To Land

### DEC-002: Treat Return-Test / PT-Transition Sessions As OOD

Sessions such as 2026-05-09's power-test day do not fit the original six daily rehab arms. They should still be preserved as body history, but should not be forced into LinUCB training.

Proposed field:

```json
{
  "session_type": "ood_power_test"
}
```

Bandit behavior:

- If `session_type` starts with `ood_`, adapter/training should skip it.
- Keep the data for retrospective analysis.
- Consider arm/schema changes only after one OOD category has at least 5 records and persists for 2+ weeks.

### DEC-003: Keep Arms Stable For Now; Reduce Context Vector First

Do not rename or replace the six arms yet. The bandit has not run, and changing arms now would move the learning target again.

Instead:

- Keep the six arms for the first working LinUCB path.
- Remove `weekday` from the initial context vector to reduce dimensionality.
- Keep `weekday` in `build_context_dict()` as optional future context, but do not feed it to `build_context_vector()` yet.

## 5. Proposed Task Changes

### Task 11: Context Vector

Initial vector should be intentionally small and explainable:

1. bias term
2. travel mode
3. yesterday score normalized to 0–1
4. days since `upper_ankle`, capped/scaled
5. days since `lower_ankle`, capped/scaled
6. days since `stretch_ankle`, capped/scaled
7. week check-in count scaled
8. last same arm completion

`weekday` remains available in the context dict but is not used in v1 vector.

### Task 12: Stepped LinUCB

Split Task 12 into five author-owned microtasks:

| Step | Function | Meaning |
|---|---|---|
| 12.1 | `compute_theta(A, b)` | Solve the linear reward model for one arm |
| 12.2 | `predict_mean(theta, x)` | Linear reward prediction |
| 12.3 | `compute_bonus(A, x, alpha)` | UCB uncertainty bonus |
| 12.4 | `select_arm(context_vec)` | Pick max `mean + bonus` |
| 12.5 | `update(arm, x, reward)` | Rank-1 update from observed reward |

Assistants may write tests and skeletons. The author writes runnable math.
